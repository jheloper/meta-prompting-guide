# 메타 프롬프팅 실행 가이드

작성일: 2026-07-02

용도: 개인 참고용. 프롬프트 생성/개선 기법 및 적용 절차.

---

## 0. 핵심 원칙

- 메타 프롬프팅 = LLM에게 프롬프트 자체를 생성/개선시키는 기법.
- 모델이 대신해주는 것: 프롬프트 문장 작성, 구조화, 예시 생성.
- 사용자가 반드시 해야 하는 것: 작업 정의, 성공 기준 정의, 검증.
- 검증 없는 메타 프롬프팅은 "그럴듯한 쓰레기" 생산 루프. 테스트 케이스가 없으면 시작하지 않는다.

---

## 1단계: 작업 정의

프롬프트를 만들기 전에 아래 4가지를 문장으로 확정한다.

| 항목 | 질문 | 예시 |
|---|---|---|
| 입력 | 모델이 받는 것은? | "사용자가 붙여넣은 회의록 텍스트" |
| 출력 | 모델이 내야 하는 것은? | "액션 아이템 목록, 담당자/기한 포함, markdown 테이블" |
| 품질 기준 | 좋은 출력의 조건은? | "회의록에 없는 내용 추가 금지, 담당자 불명확 시 '미정' 표기" |
| 실패 조건 | 절대 나오면 안 되는 것은? | "요약문만 내고 액션 아이템 누락" |

작성 요령:
- 출력 형식은 구체적으로 (형식, 길이, 언어, 톤).
- 품질 기준은 검증 가능한 형태로. "좋은 요약" (X) → "5문장 이내, 원문에 없는 사실 금지" (O)

## 2단계: 테스트 케이스 준비

- 대표 입력 3~5개 준비. 반드시 포함할 것:
  - 정상 케이스 2~3개
  - 경계 케이스 1~2개 (빈 입력, 매우 긴 입력, 형식이 어긋난 입력 등)
- 각 입력에 대한 기대 출력 또는 합격 조건 메모.
- 완벽한 정답이 없어도 됨. "이 요소가 포함되면 합격" 수준의 체크리스트면 충분.

### 2.1 테스트 케이스의 역할

테스트 케이스는 소프트웨어 테스트 개념이 아니라, 어떤 영역에서든 "좋은 결과"를 판별하는 기준 샘플이다.

| 역할 | 설명 |
|---|---|
| 채택 판단 | 생성된 프롬프트가 그럴듯한지가 아니라 실제로 동작하는지를 판단하는 유일한 근거 |
| 개선 방향 제시 | 불합격 케이스가 곧 5단계 개선 루프의 입력. 케이스가 없으면 무엇을 고칠지 알 수 없음 |
| 회귀 방지 | 수정 후 전체 케이스를 다시 돌려 기존에 되던 것이 깨지지 않았는지 확인 |
| 작업 정의 검증 | 케이스를 만들다 막히면 1단계 작업 정의가 모호하다는 신호 |

핵심: 테스트 케이스 = 대표 입력 + 합격 조건. 이 구조는 도메인과 무관하게 동일하다.

### 2.2 도메인별 예시

**예시 A: 글쓰기 (블로그 초안 작성 프롬프트)**

| 케이스 | 입력 | 합격 조건 |
|---|---|---|
| 정상 | "재택근무 생산성" + 키워드 3개 | 키워드 전부 반영, 서론-본론-결론 구조, 1500자 내외 |
| 정상 | 전문 주제 ("연말정산 변경사항") | 사실 단정 대신 "확인 필요" 표기, 출처 요구 문구 포함 |
| 경계 | 주제만 있고 키워드 없음 | 키워드를 되묻거나, 가정한 키워드를 명시하고 진행 |
| 경계 | 상충하는 요구 ("전문적이면서 초등학생도 이해") | 임의로 한쪽 선택하지 않고 우선순위를 되물음 |

**예시 B: 번역/윤문 프롬프트**

| 케이스 | 입력 | 합격 조건 |
|---|---|---|
| 정상 | 일반 이메일 영→한 | 존댓말 일관성, 원문에 없는 문장 추가 금지 |
| 정상 | 기술 용어 포함 문단 | 지정 용어집 준수 (예: "deploy"는 "배포"로 고정) |
| 경계 | 원문에 오타/비문 포함 | 오타를 임의 해석하지 않고 역주 표기 |
| 경계 | 이미 한국어인 입력 | 번역하지 않고 입력 언어 확인을 되물음 |

**예시 C: 개인 일정/계획 수립 프롬프트**

| 케이스 | 입력 | 합격 조건 |
|---|---|---|
| 정상 | "주 3회 운동, 저녁만 가능" | 제약 위반 없는 주간 계획, 요일별 표 형식 |
| 정상 | 목표 + 마감일 ("3개월 내 5kg 감량") | 주 단위 중간 목표 분해, 비현실적 목표면 경고 |
| 경계 | 제약끼리 충돌 ("매일 운동 + 주 2회 야근 + 주말 휴식") | 충돌을 지적하고 조정안 제시 |
| 경계 | 정보 부족 ("운동 계획 짜줘"만) | 현재 수준/가용 시간을 되물음 |

**예시 D: 학습/설명 프롬프트**

| 케이스 | 입력 | 합격 조건 |
|---|---|---|
| 정상 | "복리 개념 설명해줘" | 정의 + 일상 비유 + 계산 예시 1개 포함 |
| 정상 | 후속 질문 ("그럼 단리와 차이는?") | 앞 설명과 일관된 용어 사용 |
| 경계 | 잘못된 전제 포함 질문 ("복리는 원금이 줄어드는 거지?") | 전제를 교정한 후 설명 |
| 경계 | 범위가 너무 넓은 질문 ("경제학 알려줘") | 하위 주제 목록 제시 후 선택 유도 |

### 2.3 합격 조건 작성 요령

- 검증 가능한 형태로: "자연스러운 번역" (X) → "존댓말 일관, 원문에 없는 내용 추가 금지" (O)
- 경계 케이스의 합격 조건은 대부분 둘 중 하나: (a) 되묻는다, (b) 가정을 명시하고 진행한다. 어느 쪽을 원하는지 1단계에서 미리 결정.
- 합격 조건 3~5개면 충분. 10개 넘으면 케이스를 분리할 것.

## 3단계: 메타 프롬프트 작성 (생성)

모델에게 프롬프트를 만들게 하는 지시문. 아래 템플릿 사용.

```
당신은 프롬프트 엔지니어다. 아래 작업을 수행하는 시스템 프롬프트를 작성하라.

## 작업 정의
- 입력: {입력 설명}
- 출력: {출력 형식, 길이, 언어}
- 품질 기준: {검증 가능한 기준 목록}
- 금지 사항: {실패 조건}

## 프롬프트 요구사항
- 대상 모델: {Claude / GPT / Gemini}
- 역할(persona) 정의 포함
- 출력 형식을 명시적으로 지정
- few-shot 예시 {N}개 포함
- 애매한 입력 처리 방법 명시 (예: 정보 부족 시 되묻기 vs 가정 명시)

## 출력
완성된 시스템 프롬프트만 출력. 설명이나 서두 없이.
```

작성 요령:
- "프롬프트만 출력" 지시 필수. 없으면 해설이 섞여 나옴.
- few-shot 예시 개수를 지정. 미지정 시 과다하게 생성되는 경향.
- 대상 모델을 명시하면 해당 모델 관례(XML 태그 등)에 맞춰 생성됨.

---

## 4단계: 검증

1. 생성된 프롬프트를 실제 대상 모델에 설정.
2. 2단계 테스트 케이스 전체 실행.
3. 케이스별 합격/불합격 기록. 불합격은 실패 양상을 구체적으로 메모.

```
케이스 1 (정상): 합격
케이스 2 (정상): 합격
케이스 3 (긴 입력): 불합격 - 후반부 액션 아이템 누락
케이스 4 (빈 입력): 불합격 - 되묻지 않고 가상의 내용 생성
```

주의:
- 메타 프롬프트를 실행한 모델과 다른 모델에서 검증할 것 (자기 채점 편향 방지).
- 1회 실행으로 판단하지 말 것. 비결정성 때문에 애매한 케이스는 2~3회 실행.

---

## 5단계: 개선 루프

불합격 케이스를 피드백으로 재생성한다.

```
아래 시스템 프롬프트가 특정 입력에서 실패한다. 실패를 해결하도록 프롬프트를 수정하라.
다른 동작은 유지할 것.

## 현재 프롬프트
{프롬프트 전문}

## 실패 사례
- 입력: {실패한 입력}
- 실제 출력: {잘못된 출력}
- 기대 출력: {기대했던 결과}

## 출력
수정된 프롬프트 전문만 출력.
```

루프 운영 규칙:
- 한 번에 하나의 실패 유형만 수정 (여러 개 동시 수정 시 회귀 발생).
- 수정 후 전체 테스트 케이스 재실행 (회귀 검사).
- 3회 루프 후에도 해결 안 되면 프롬프트 수정이 아니라 1단계 작업 정의로 복귀. 정의가 모호한 경우가 대부분.
- 버전 관리: 프롬프트에 버전과 날짜 주석 (`v3, 2026-07-02`). 이전 버전으로 롤백 가능하게.

## 6단계: 멀티 서비스 적용 (Claude / GPT / Gemini)

하나의 프롬프트를 여러 서비스에 배포할 때의 변환 절차.

### 6.1 마스터 프롬프트 원칙
- 특정 서비스 문법에 의존하지 않는 마스터 버전을 유지.
- 서비스별 버전은 마스터에서 변환하여 생성. 서비스별 버전을 직접 수정하지 않는다 (드리프트 방지).

### 6.2 서비스별 차이 요약

| 항목 | Claude | GPT (CustomGPT) | Gemini (Gems) |
|---|---|---|---|
| 구조화 선호 | XML 태그 (`<rules>`) | markdown 헤더 | markdown 헤더 |
| 시스템 프롬프트 위치 | Projects 지침 / API system | Instructions 필드 | Instructions 필드 |
| 지시 준수 성향 | 명시적 규칙에 충실 | 간결한 지시 선호, 장문 시 후반 무시 경향 | 예시 기반 지시에 강함 |
| 파일 참조 | Projects 지식 / repo 연동 | Knowledge 업로드 | 파일 첨부 |

### 6.3 변환도 메타 프롬프팅으로 처리

```
아래 마스터 프롬프트를 {대상 서비스}의 Instructions 필드에 맞게 변환하라.

변환 규칙:
- 의미와 규칙은 100% 보존
- {대상 서비스} 관례에 맞는 구조로 재구성 (예: XML 태그 → markdown 헤더)
- Instructions 글자 수 제한 {N}자 이내로 압축
- 외부 파일 참조 지시는 제거하고 핵심 규칙을 인라인

## 마스터 프롬프트
{전문}
```

### 6.4 멀티 서비스 검증
- 동일 테스트 케이스를 3개 서비스 모두에서 실행.
- 서비스별 실패는 서비스별 변환 규칙에만 반영. 마스터는 서비스 중립 실패만 반영.

## 7. 체크리스트 (요약)

- [ ] 입력/출력/품질 기준/실패 조건 문장으로 확정
- [ ] 테스트 케이스 3~5개 + 합격 조건
- [ ] 메타 프롬프트로 초안 생성 (프롬프트만 출력하도록 지시)
- [ ] 대상 모델에서 전체 케이스 검증 (생성 모델과 분리)
- [ ] 실패 사례 1건씩 피드백 루프, 매회 회귀 검사
- [ ] 3회 루프 실패 시 작업 정의 재검토
- [ ] 마스터 프롬프트 유지, 서비스별 버전은 변환으로 생성
- [ ] 버전/날짜 기록

## 부록: 흔한 실패 패턴

| 증상 | 원인 | 대응 |
|---|---|---|
| 생성된 프롬프트가 장황함 | 길이 제약 미지정 | "N자 이내" 명시 |
| 그럴듯하지만 실전에서 실패 | 테스트 케이스 없이 채택 | 2단계로 복귀 |
| 수정할수록 다른 부분이 깨짐 | 여러 실패 동시 수정 | 한 번에 하나, 회귀 검사 |
| 서비스마다 동작이 다름 | 서비스별 버전 직접 수정 | 마스터 단일 소스 유지 |
| 모델이 자기 프롬프트를 후하게 평가 | 자기 채점 편향 | 검증은 다른 모델/수동으로 |
